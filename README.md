Ниже — ТЗ и MVP-план для проекта **«Трекер подписок и регулярных платежей»**. Проект хороший для full-stack портфолио, потому что в нём есть авторизация, CRUD, даты, напоминания, аналитика, экспорт, фоновые задачи и понятная бизнес-логика.


# Состояние реализации

Реализованы **MVP 0: Setup**, **MVP 1: Auth** и **MVP 2: CRUD подписок**.
Проект содержит базовый монорепозиторий с frontend, backend, PostgreSQL,
Prisma, Docker Compose, авторизацией и управлением подписками.

Технические решения MVP 0:

* линтер настроен в строгом type-aware режиме для TypeScript;
* frontend styling зафиксирован на CSS Modules;
* Tailwind CSS удалён из frontend-конфигурации, чтобы не смешивать подходы.

Технические решения MVP 1:

* регистрация, вход, выход и `GET /auth/me`;
* пароль хранится в bcrypt-хеше;
* JWT хранится в httpOnly cookie;
* `/dashboard` защищён server-side проверкой текущего пользователя.

Технические решения MVP 2:

* backend endpoints `GET/POST/PATCH/DELETE /subscriptions`;
* все операции с подписками фильтруются по текущему `user_id`;
* сумма хранится в PostgreSQL `Decimal(12, 2)`, API возвращает её строкой;
* frontend страницы `/subscriptions`, `/subscriptions/new`,
  `/subscriptions/[id]`, `/subscriptions/[id]/edit`;
* необязательная категория в форме пока автосоздаётся по названию без отдельного
  интерфейса управления категориями.

Структура:

```text

  web/        # Next.js App Router frontend
  api/        # NestJS API + Prisma
packages/
```

# 1. Краткое описание проекта

**Subscription Tracker** — веб-сервис, в котором пользователь может вести список регулярных платежей: подписки, аренда сервера, домены, интернет, мобильная связь, кредиты, рассрочки и другие периодические списания.

Сервис помогает пользователю:

* видеть ближайшие списания;
* понимать, сколько денег уходит в месяц;
* группировать расходы по категориям;
* получать напоминания перед списанием;
* анализировать расходы по месяцам;
* экспортировать данные в CSV или PDF.

# 2. Цель проекта

Создать личный финансовый мини-сервис для контроля регулярных расходов.

Главная ценность MVP:
**пользователь добавляет свои регулярные платежи и видит, когда и сколько будет списано в ближайшее время.**

# 3. Целевая аудитория

Основная аудитория:

* пользователи с большим количеством подписок;
* фрилансеры и разработчики, которые платят за серверы, домены, SaaS-сервисы;
* небольшие команды, которым нужно отслеживать регулярные расходы;
* люди, которые хотят контролировать личный бюджет.

# 4. Основные пользовательские сценарии

## Сценарий 1. Регистрация

Пользователь заходит на сайт, регистрируется через email и пароль, после чего попадает в личный кабинет.

## Сценарий 2. Добавление подписки

Пользователь нажимает «Добавить подписку», вводит:

* название;
* стоимость;
* валюту;
* дату следующего списания;
* периодичность;
* категорию;
* описание.

После сохранения подписка появляется в списке и календаре.

## Сценарий 3. Просмотр ближайших списаний

Пользователь открывает dashboard и видит:

* сколько будет списано в этом месяце;
* ближайшие платежи;
* подписки, которые скоро спишутся;
* общую сумму регулярных расходов.

## Сценарий 4. Редактирование подписки

Пользователь может изменить цену, дату списания, категорию, периодичность или отключить подписку.

## Сценарий 5. Напоминание

Система отправляет уведомление за 1, 3 или 7 дней до списания.

## Сценарий 6. Аналитика

Пользователь видит статистику расходов по месяцам и категориям.

## Сценарий 7. Экспорт

Пользователь может скачать список подписок или отчёт за месяц в CSV/PDF.

# 6. Функциональные требования

## 6.1. Регистрация и авторизация

Пользователь должен иметь возможность:

* зарегистрироваться по email и паролю;
* войти в аккаунт;
* выйти из аккаунта;
* оставаться авторизованным после перезагрузки страницы;
* видеть только свои данные.

Минимальные требования:

* пароль хранится в хешированном виде;
* нельзя зарегистрировать два аккаунта на один email;
* при неправильном логине или пароле показывается ошибка;
* защищённые страницы недоступны без авторизации.

Дополнительно после MVP:

* восстановление пароля;
* подтверждение email;
* OAuth через Google;
* двухфакторная авторизация.

# 6.2. CRUD подписок

Пользователь должен иметь возможность создать подписку.

Поля подписки:

| Поле                       |         Тип | Обязательное | Пример        |
| -------------------------- | ----------: | -----------: | ------------- |
| Название                   |      string |           да | Netflix       |
| Цена                       |      number |           да | 9.99          |
| Валюта                     | enum/string |           да | USD           |
| Категория                  |    relation |          нет | Entertainment |
| Дата следующего списания   |        date |           да | 2026-08-01    |
| Периодичность              |        enum |           да | monthly       |
| Описание                   |      string |          нет | Family plan   |
| Активна/неактивна          |     boolean |           да | true          |
| Напоминание включено       |     boolean |          нет | true          |
| За сколько дней напоминать |      number |          нет | 3             |

Периодичность:

* daily;
* weekly;
* monthly;
* quarterly;
* yearly;
* custom — можно добавить позже.

Пользователь должен иметь возможность:

* создать подписку;
* посмотреть список подписок;
* открыть одну подписку;
* отредактировать подписку;
* удалить подписку;
* отключить подписку без удаления.

Важно:
**удаление и отключение — разные действия.**

Удаление полностью убирает подписку.
Отключение оставляет её в истории, но она больше не учитывается в ближайших списаниях.

# 6.3. Список подписок

На странице подписок пользователь видит таблицу или карточки.

Минимальные поля:

* название;
* цена;
* валюта;
* категория;
* следующая дата списания;
* периодичность;
* статус.

Фильтры:

* по категории;
* по активности;
* по валюте;
* по дате ближайшего списания.

Сортировка:

* по дате списания;
* по цене;
* по названию.

Для MVP достаточно:

* список;
* поиск по названию;
* фильтр по категории;
* сортировка по ближайшей дате.

# 6.4. Календарь ближайших списаний

Пользователь должен видеть ближайшие списания в календарном виде.

Минимальная версия:

* список списаний на ближайшие 30 дней;
* группировка по датам;
* сумма списаний за выбранный период.

Хорошая версия:

* месячный календарь;
* на каждом дне отображаются платежи;
* при клике на день открывается список списаний;
* отображается сумма за день.

Пример:

```text
5 августа
- Netflix — $9.99
- Spotify — $5.99

12 августа
- VPS Server — €12.00
```

# 6.5. Категории расходов

Пользователь должен иметь возможность группировать подписки по категориям.

Базовые категории можно создать автоматически:

* Entertainment;
* Work;
* Hosting;
* Internet;
* Education;
* Health;
* Other.

Пользователь может:

* создать категорию;
* переименовать категорию;
* удалить категорию;
* назначить категорию подписке.

При удалении категории подписки не удаляются.
Они переносятся в `Other` или остаются без категории.

# 6.6. Напоминания

Пользователь должен иметь возможность включить напоминания перед списанием.

Варианты:

* за 1 день;
* за 3 дня;
* за 7 дней;
* кастомное значение — позже.

Для MVP можно сделать простую реализацию:

* напоминания отображаются внутри приложения;
* email-уведомления можно добавить отдельным этапом.

Варианты уведомлений:

| Тип                |         MVP |      После MVP |
| ------------------ | ----------: | -------------: |
| In-app уведомления |          да |             да |
| Email              | опционально |             да |
| Telegram bot       |         нет | можно добавить |
| Push               |         нет | можно добавить |

Логика:

* система каждый день проверяет активные подписки;
* если дата списания скоро, создаётся уведомление;
* пользователь видит уведомление на dashboard.

# 6.7. Статистика по месяцам

Пользователь должен видеть аналитику расходов.

Минимальная статистика:

* общая сумма в месяц;
* сумма по категориям;
* количество активных подписок;
* самая дорогая подписка;
* ближайшее крупное списание.

Графики:

* расходы по месяцам;
* расходы по категориям;
* динамика общей суммы.

Для MVP можно считать статистику на основе текущих активных подписок.

После MVP лучше добавить историю платежей, чтобы статистика была точнее.

# 6.8. Экспорт в CSV/PDF

Пользователь должен иметь возможность экспортировать данные.

## CSV


## PDF

PDF-отчёт может содержать:

* период отчёта;
* список подписок;
* общую сумму;
* расходы по категориям;
* ближайшие списания.

Для MVP лучше сначала сделать CSV.
PDF можно добавить позже, потому что он требует больше работы с форматированием.

# 7. Нефункциональные требования

## Безопасность

* пароли хранятся только в хешированном виде;
* все пользовательские данные доступны только владельцу;
* API защищён авторизацией;
* нельзя получить чужую подписку по ID;
* валидация данных на backend;
* rate limit на login/register — желательно.

## Производительность

Для MVP достаточно:

* dashboard открывается меньше чем за 1 секунду на небольшом количестве данных;
* список подписок поддерживает хотя бы 100–500 записей на пользователя;
* статистика считается без заметной задержки.

## Надёжность

* ошибки API должны возвращать понятные сообщения;
* удаление подписки требует подтверждения;
* фоновые напоминания не должны дублироваться каждый день бесконечно.

## UX

* добавление подписки должно занимать не больше 1 минуты;
* dashboard должен сразу показывать главные цифры;
* формы должны иметь валидацию;
* пустые состояния должны объяснять, что делать дальше.

# MVP 6. In-app напоминания

Цель: пользователь видит уведомления о скорых списаниях.

Фичи:

* настройка `reminder_enabled`;
* настройка `reminder_days_before`;
* генерация уведомлений;
* список уведомлений;
* отметка уведомления прочитанным.

Backend endpoints:

```http
GET /notifications
PATCH /notifications/:id/read
```

Фоновая задача:

```text
Каждый день:
1. Найти активные подписки.
2. Проверить дату следующего списания.
3. Если списание через N дней — создать уведомление.
4. Не создавать дубликат, если уведомление уже есть.
```

Готово, если:

* за N дней до списания появляется уведомление;
* уведомление видно на dashboard;
* уведомление можно отметить прочитанным;
* уведомления не дублируются.

* годовая оценка расходов.

# 14. Бизнес-логика периодичности

Нужно уметь вычислять следующую дату списания.

Пример:

```text
monthly:
  2026-08-05 -> 2026-09-05

yearly:
  2026-08-05 -> 2027-08-05

weekly:
  2026-08-05 -> 2026-08-12
```

Проблемные случаи:

```text
31 января + 1 месяц = 28 февраля или 29 февраля
```

Для MVP можно принять правило:

> Если в следующем месяце нет такого дня, списание переносится на последний день месяца.

Например:

```text
31 января -> 28 февраля
31 марта -> 30 апреля
```
